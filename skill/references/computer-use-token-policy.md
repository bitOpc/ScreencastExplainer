# Computer Use 省 Token 策略

本文档是 **Agent 操作建议**，不是 CLI 参数，也**没有**对应的 Python 模式开关。

代码里不会出现 `精简` / `常态` / `浪费` 枚举；Agent 在步骤 6（校准滚动）与步骤 7（录屏 + 并行操作）时，按用户时长与复杂度自行选择下列策略。

## 背景

Computer Use 每次 `capture` / 截图会消耗大量 context token（Hermes 文档量级：约 1500 token/张，且会保留最近若干张）。  
录屏阶段画面由 `record_window.py` + `screencapture` 负责，**不必**为确认滚动而频繁截图。

## 三档策略（Agent 自选）

| 策略 | 适用场景 | capture 频率 | 全流程 token 粗估* |
|------|----------|--------------|-------------------|
| **精简** | 短稿（≤3 分钟）、滚动简单、窗口稳定 | 校准阶段每段 1 次确认；录屏中**不再** capture | 约 8–15 万 |
| **常态** | 默认；中等时长、需边滚边确认 | 校准每段 1–2 次；录屏中仅在段落切换点 capture | 约 15–30 万 |
| **浪费** | 复杂 UI、多面板切换、反复试错 | 几乎每步 capture | 50 万+ |

\* 含脚本撰写、分段、旁白生成与合成；实际因模型与平台而异，仅作量级参考。

## 精简模式（推荐默认）

1. 步骤 6：对每种滚动策略最多测 1 次，用 `list_windows` / 文本状态判断即可，不必每张都截图。
2. 步骤 7：录屏开始后依赖 `segments.json` 时间轴执行 scroll，**录屏期间禁止**为「看看画面」而 capture。
3. 录屏结束后用 `ingest_capture.py` 校验时长；失败再针对性重录，而不是录屏中反复截图。

## 常态模式

- 步骤 6：每段旁白绑定前 capture 一次，确认 page_target 对齐。
- 步骤 7：每 2–3 段 capture 一次，防止滚动失效。
- 仍禁止「每步操作都截图」。

## 浪费模式（避免）

- 录屏全程高频 capture → token 暴涨，且与真录屏路径重复。
- 仅在用户明确要求「录屏过程中实时盯画面」且接受高 token 消耗时考虑。

## 与代码的对应关系

| 概念 | 是否在代码中 |
|------|----------------|
| `record_window.py` 单窗口录屏 | ✅ 有 |
| `doctor.py` 依赖检查 | ✅ 有 |
| 精简 / 常态 / 浪费 | ❌ 无；见本文档，由 Agent 执行时自律 |

用户若问「有没有省 token 模式」，回答：**没有开关，按本文档的「精简」策略执行即可。**
